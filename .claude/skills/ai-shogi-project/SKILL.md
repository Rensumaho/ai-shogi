# AI将棋プロジェクト — プロジェクト概要と設計方針

## プロジェクトの目的

LLM（Claude、GPT、Gemini等）と対局できる将棋アプリケーションを開発する。

## システム構成

```
┌─────────────┐       MCP通信        ┌──────────────────┐
│  Cursor IDE  │ ◄──────────────────► │  将棋アプリ       │
│  (LLMクライアント)│                    │  (MCPサーバー)    │
└─────────────┘                      └──────────────────┘
       │                                     │
       │ ユーザーが「将棋をして」と指示          │ 盤面をGUIで表示
       │ LLMが次の手を決定                    │ ユーザーがGUI上で駒を操作
       ▼                                     ▼
  LLMが盤面状況を把握し               リアルタイムで駒が動く
  最善手をMCP経由で送信
```

## 利用フロー

1. ユーザーが将棋アプリを起動する（MCPサーバーとして動作開始）
2. CursorのMCP設定に将棋アプリを登録する
3. ユーザーがCursorに「将棋をして」と指示する
4. Cursor（LLM）がMCP経由で将棋アプリに接続し、対局を開始する
5. LLMの手番：LLMが盤面を分析し、次の指し手を決定 → MCP経由で将棋アプリに送信 → アプリ画面上で駒がリアルタイムに動く
6. ユーザーの手番：ユーザーが将棋アプリのGUI上で自分の駒を操作する
7. 将棋アプリがユーザーの操作をMCP経由でCursorに通知する
8. 5〜7を繰り返し、対局が終了するまで続ける

## ユーザー操作の検知方式（手番交代の実現）

MCPはクライアント→サーバーへの一方向リクエストが基本であり、サーバーからクライアントへの能動的な通知は標準的ではない。そのため「ユーザーが駒を動かしたことをLLMが検知する」部分に工夫が必要。

### 採用方式：ブロッキングツール方式（主） + ユーザー仲介方式（フォールバック）

**主方式 — ブロッキングツール（`wait_for_user_move`）:**

`wait_for_user_move` ツールを呼ぶと、ユーザーが操作を完了するまでレスポンスを返さない。ユーザーが指し手を完了した時点で結果を返す。

```
1. LLM → make_move("7六歩") → 盤面に即反映
2. LLM → wait_for_user_move() → サーバーが応答を保留（ブロック）
3. ユーザーがGUI上で駒を操作
4. サーバーがユーザーの操作完了を検知 → レスポンス返却（"3四歩"）
5. LLM → 盤面を分析 → 次の手を決定 → 1に戻る
```

**フォールバック — ユーザー仲介方式:**

ブロッキング方式がタイムアウト等で機能しない場合に備える。ユーザーがGUIで指した後、Cursorのチャットに「指した」等と入力し、LLMが `get_board_state` で盤面を取得して次の手を決定する。

```
1. LLM → make_move("7六歩") → 「あなたの番です。指し終わったら教えてください」
2. ユーザーがGUIで駒を動かす
3. ユーザーがCursorに「指した」と入力
4. LLM → get_board_state() → 盤面取得 → 次の手を決定 → 1に戻る
```

### 実装上の注意

- 両方式を共存させ、環境やタイムアウト状況に応じて切り替えられるようにする
- `wait_for_user_move` のタイムアウト時間は設定可能にする
- タイムアウト発生時はエラーではなく「まだユーザーが操作中」のステータスを返す

## 設計上の重要ポイント

- **MCPサーバーとしての将棋アプリ**: 将棋アプリ自体がMCPサーバーとして機能し、Cursorから呼び出し可能なツール群を提供する
- **リアルタイム双方向通信**: LLMの指し手がアプリ画面に即座に反映され、ユーザーの操作もLLM側に即座に伝わる
- **盤面状態の管理**: 将棋のルール（駒の動き、成り、持ち駒、王手判定等）をアプリ側で正しく管理する
- **LLM非依存**: Claude、GPT、Gemini等、どのLLMからでも利用できるよう、MCP標準プロトコルに準拠する

## MCP ツール設計の方向性

将棋アプリが提供すべきMCPツール：

- `start_game` — 新しい対局を開始
- `get_board_state` — 現在の盤面状態を取得
- `get_legal_moves` — 合法手一覧を取得
- `make_move` — 指し手を実行（駒を動かす）
- `wait_for_user_move` — ユーザーの操作完了を待つ（ブロッキング）
- `get_game_status` — 対局状態（進行中、詰み、千日手等）を取得
- `get_move_history` — 棋譜（指し手の履歴）を取得

## 技術スタック

- **Next.js 15** (App Router) + **TypeScript** — Web GUI + API
- **shogiops** — 将棋ルールエンジン（合法手生成、SFEN、詰み判定）
- **@modelcontextprotocol/sdk** + **zod** — MCPサーバー（stdio transport）

## セットアップ手順

1. `npm install` で依存関係をインストール
2. `npm run dev` で Next.js アプリを起動（将棋GUI + API）
3. `.cursor/mcp.json` に MCP サーバー設定が済んでいるので、Cursor が自動的に MCP サーバーを起動
4. ブラウザで `http://localhost:3001` を開き、将棋盤を表示
5. Cursor に「将棋をして」と指示して対局開始

## エージェントへの注意事項

- 常にこのプロジェクトの最終ゴール（LLMと将棋を打てるMCPベースのアプリ）を意識して作業すること
- 将棋のルールを正確に実装すること（不正な手を許可しない）
- MCPプロトコルへの準拠を最優先とし、特定のLLMに依存した実装を避けること
- UIはリアルタイム性を重視し、操作の即時反映を心がけること
